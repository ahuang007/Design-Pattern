
# Facade 模式

## 问题

* 举个生活中的小例子，大凡开过学或者毕过业的的都会体会到这样的一种郁闷：你要去 n 个地方办 n 个手续。但是实际上我们需要的就是一个最后一道手续的证明而已，对于前面的手续是怎么办的、到什么地方去办理我们都不感兴趣。

* 实际上在软甲系统开发中也经常会遇到这样的问题，可能你实现了一些借口（模块），而这些接口（模块）都分布在几个类中（比如 A 和 B、C、D）：A 中实现了一些接口，B中实现了一些接口（或者 A 代表一个独立模块，B、C、D 代表另一些独立模块）。然后你的客户程序员（使用你设计的开发人员）只有很少的人要知道你的不同接口到底是在那一个类中实现的，绝大多数只是想简单的组合你的 A-D 的类的接口，他并不想知道这些接口在哪里实现的。

* 这里的客户程序员就是上面生活中想办理手续的郁闷的人！在现实生活中我们可能可以找到一个人代理所有的事情就可以解决你的问题（你只要维护和他的简单的一个接口而已了！），在软件系统设计开发中我们可以通过一个叫 Facade 的模式来解决上面的问题。

## 代码说明

* Facade 模式实现很简单，多余的解释完全是没有必要的

## 讨论

* Facade 模式在高层提供了一个统一的接口，解耦了系统。设计模式 中海油另一种模式 Mediator 也和 Facade 有类似的地方。但是 Mediator 主要目的是对象间的访问的解耦（通讯时候的协议）。
